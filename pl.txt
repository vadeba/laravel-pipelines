CI/CD Пайплайны для приложения Laravel 
1. Добавить тесты – покрытие 50% 
2. Создайте конфигурационные файлы для разных сред 
• 
.env.dev, .env.uat, .env.prod со значениями переменных, описанных ниже 
• 
Обязательные 
переменные: APP_NAME, APP_ENV, APP_DEBUG, APP_URL, DB_CONNECTION, D
B_HOST, DB_PORT, DB_DATABASE, DB_USERNAME, DB_PASSWORD, APP_KEY (ma
y be left blank with a note). 
• 
Основной .env файл не должен попасть в репозиторий; Удостоверьтесь, что 
.gitignore настроен. 
• 
Добавьте .env.ci файл для запуска пайплайна (e.g., SQLite in-memory, no debug). 
3. Создайте репозиторий GitHub или GitLab 
4. Помимо master (main) создайте ветки development (dev/develop) и uat (qa) 
5. Создайте пайплайн в репозитории (GitHub Actions или GitLab CI/CD) 
Пайплайн должен вызываться при любом обновлении (push, PR/MR) трех 
долгоживущих веток dev, qa, master. Шаги пайплайна: 
a. Тесты 
Вызовите исполнение тестов Laravel 
Gate: пайплайн завершается с ошибкой если покрытие < 50% или какой-
то из тестов завершился ошибкой 
b. Статический анализ кода 
Используйте PHPStan (Larastan) 
Пайплайн завершается с ошибкой при любом нарушении, найденном 
при анализе кода (при любой ошибке, не предупреждении) 
c. Linting  
Используйте Laravel Pint (или PHP_CodeSniffer) с пресетом PSR-12 / 
Laravel 
Запуск в “test” режиме – пайплайн завершается с ошибкой при любом 
нарушении правила линтера, не включайте авто-исправление кода 
(Linter должен форматировать код во время пайплайна при пуше в 
любую другую ветку) 
d. Симуляция деплоя 
Запускается для долгоживущих веток: 
o develop → use .env.dev
o uat → use .env.uat 
o main / master → use .env.prod 
Скопируйте соотевтствующий .env файл как .env в раннере пайплайна и 
выведите сообщение “Deploying to [ENV] with .env.[env]”. 
Этот шаг пайплайна выполняется только если предыдущие шаги успешно 
завершились. 
Для продуктовых веток (mian/master) добавьте дополнительное условие – 
ручной аппрув шага.  (см. GitHub Environments с обязательным reviewer, или 
GitLab’s when) 
*Доп задание – После выполнения пайплайна необходимо отправить сообщение 
выбранным пользователям (maintainers) с результатом пайплайна 
 
Для проверки 
1. Ссылка на репозиторий. 
2. Файл с конфигурацией пайплайна (.github/workflows/ci.yml or .gitlab-ci.yml). 
3. Файлы с конфигами для сред: .env.dev, .env.uat, .env.prod, .env.ci. 
4. PIPELINE.md с описанием пайплайна 
5. 3 скриншота: успешный пайплайн, нарушение в шаге тестирования, 
нарушение в шаге linter.
